## Outline

Goal: act as a research helper for another Kite agent. Given candidate frameworks, hosting targets, dependency questions, or AI SDK options, verify the official guidance and record the results in `research.md` without becoming a separate workflow stage.

Execution steps:

1. Run `{SCRIPT}` from repo root **once** (combined `--json --paths-only` mode / `-Json -PathsOnly`). Parse the minimal JSON payload fields:
   - `REPO_ROOT`
   - `FEATURE_DIR`
   - `FEATURE_SPEC`
   - `IMPL_PLAN`
   - `TASKS`
   - If JSON parsing fails, abort and instruct the user to re-run `__KITE_COMMAND_SPECIFY__` or verify the feature-branch environment.
   - For single quotes in args like "I'm Groot", use escape syntax: e.g. 'I'\''m Groot' (or double-quote if possible: "I'm Groot").

2. Read the current planning context:
   - Required: `FEATURE_SPEC`
   - Optional: `FEATURE_DIR/design.md`, `FEATURE_DIR/discovery.md`, `kite.config.yml`, `.kite/memory/constitution.md`, existing `FEATURE_DIR/research.md`
   - If `FEATURE_SPEC` is missing, instruct the user to run `__KITE_COMMAND_SPECIFY__` first.

3. Treat the calling agent as the owner of user interaction:
    - Default behavior: do **not** start a new preference interview.
    - Use the caller's prompt, `kite.config.yml`, and current artifacts as your input set.
    - Only ask the user a question yourself if the caller explicitly told you to collect a specific missing answer.

4. Build a research task list from the caller's brief. Common task types:
    - Verify the current stable version of a framework or dependency from official docs
    - Compare shortlisted frameworks against hosting or operational constraints
    - Verify official quickstarts, install targets, and release notes
    - For AI SDKs or agent frameworks, check official support for MCP, skills, starter kits, templates, or agent best practices

5. Research loop for every shortlisted or selected tool:
    - **Official docs first.** Use the vendor's official docs, install guide, release notes, or package registry page before any blog post or secondary source.
    - Verify the **current stable version** or the official install target.
    - Never recommend `latest` or floating dependency versions; record concrete verified versions or project-approved ranges.
    - Capture the canonical quickstart or installation URL.
   - Record compatibility notes with the rest of the chosen stack.
   - For AI SDKs / agent frameworks, explicitly check the official docs for:
     - MCP servers or official MCP integrations
     - Skills, templates, starter kits, or agent-specific guidance assets
     - Best-practice guidance for AI agents using the framework
   - Examples worth checking when relevant: Mastra, Google ADK, Vercel AI SDK, LangChain. Do **not** assume support exists until you verify it in the official docs.
   - If the official docs do not publish a version, use the official package registry or official GitHub releases page. If the version still cannot be verified, write `NEEDS RESEARCH` instead of guessing.

6. Write or refresh `FEATURE_DIR/research.md` with this structure:

   ```markdown
   # Technical Research Brief

   **Generated by:** kite.research
   **Date:** <ISO-8601 date>

   ## What this means in plain English

   > One short paragraph that the parent agent can reuse. Explain the recommended stack direction, the hosting fit, and whether an AI SDK is needed.

   ## 1. Caller context

   | Topic | Decision | Why |
   |-------|----------|-----|
   | Backend | ... | ... |
   | Frontend | ... | ... |
   | Hosting | ... | ... |
   | Budget | ... | ... |
   | AI SDK | ... | ... |

   ## 2. Recommended stack

   ### 2.1 Application / backend

   - ...

   ### 2.2 Frontend / client

   - ...

   ### 2.3 Hosting

   - ...

   ### 2.4 AI SDK / agent framework

   - State explicitly when no AI SDK is needed.

   ## 3. Current version guidance

   | Tool | Recommended version | Source | Notes |
   |------|---------------------|--------|-------|

   ## 4. Official docs and agent support

   | Tool | Official docs | MCP / skill / agent support | Notes |
   |------|---------------|-----------------------------|-------|

   ## 5. Rejected options

   - ...

   ## 6. Open questions for the caller

   - [ ] ...
   ```

   Rules for the brief:
    - Use real source URLs.
    - Every version entry must cite the official source used to verify it.
    - No install command or dependency recommendation may use `latest` or a floating version.
    - The AI SDK section must explicitly say either `Not needed for this feature` or name the chosen SDK / framework.
   - The agent-support table must explicitly say whether MCP / skills guidance was found, not just leave the cell vague.

7. Validation before finishing:
   - Every recommended version in `research.md` has an official source URL.
   - The hosting recommendation reflects the caller's stated cost or operations preference.
   - The AI SDK section explicitly says whether an AI SDK is needed.
   - For AI SDKs / agent frameworks, the MCP / skills check is explicitly documented.
   - If a version or capability could not be verified, it is marked `NEEDS RESEARCH` instead of being guessed.

8. Report completion to the calling agent:
   - Path to `research.md`
   - Verified backend / frontend / hosting direction
   - Whether an AI SDK is needed
   - Any MCP / skill findings that affect the next phase
   - Any remaining open questions
   - A short list of official source URLs worth keeping in the plan

Behavior rules:

- Prefer official docs over memory or secondary sources.
- Keep suggestions pragmatic; optimize for stack fit, maintenance burden, and AI-agent usability.
- If the caller already provided a clear preference, verify it instead of arguing with it.
- If the feature is not AI-related, do not force an AI SDK into the plan.
- If the current versions cannot be verified, stop short of pinning them.
- Do **not** update `.kite/state.yml`; the parent agent owns workflow state.
