## User Input

```text
$ARGUMENTS
```

You **MUST** consider the user input before proceeding (if not empty). The user input is optional guidance for browser validation (for example, "only test auth" or "skip screenshots"). Your job is to validate the connected app in a browser, write `FEATURE_DIR/browser-report.md`, and classify issues without editing product code.

## Outline

This command is a validation helper for `kite.frontend` only. It is not a product-code implementation stage, and other agents should consume `browser-report.md` instead of invoking `kite.browser` directly.

### Hard rules for this command

1. **Never edit product code.** This command may read files, run the app, use browser tooling, and write `browser-report.md`, but it must not modify application source, tests, contracts, or task checkboxes.
2. **Use existing artifacts first.** Read `contract.md`, `design.md`, `design-system.md`, `quickstart.md`, `tasks.md`, and `research.md` when present before guessing any flow, route, or command.
3. **Prefer the project's documented run flow.** Reuse the startup or verification commands already recorded in `quickstart.md`, `kite.config.yml`, or the existing tasks. Ask one question only if the browser run command is still unclear.
4. **Classify every issue.** Each failure must be labeled `[frontend]`, `[backend]`, `[contract]`, `[ops]`, or `[unknown]` so the caller knows where to send the fix.
5. **Frontend-only subagent.** `kite.browser` MUST only run as a focused subagent invoked by `kite.frontend`. It reports evidence back to the parent frontend agent through `browser-report.md` instead of becoming a broad debugging session or a standalone QA workflow. If invoked outside `kite.frontend`, refuse with a one-line message pointing back to `kite.frontend`.
6. **Stay honest about blockers.** If the app cannot start or a dependency is missing, record the blocker in the report and stop short of inventing a workaround.

### Step 1 — Read existing artifacts

Run `{SCRIPT}` from the repo root and parse `FEATURE_DIR`. Use that active feature directory for every artifact path. Do not guess by sorting `specs/` directories.

Required:

- `FEATURE_DIR/tasks.md`

Optional but preferred:

- `FEATURE_DIR/contract.md`
- `FEATURE_DIR/design.md`
- `FEATURE_DIR/design-system.md`
- `FEATURE_DIR/quickstart.md`
- `FEATURE_DIR/research.md`
- `kite.config.yml`
- existing `FEATURE_DIR/browser-report.md`

If `tasks.md` is missing, abort and tell the user to run `kite.tasks` first.

### Step 2 — Determine the browser validation flow

1. Identify the primary flows to validate from `tasks.md`, `design.md`, and `contract.md`.
2. Identify how to run or connect to the app from `quickstart.md`, `kite.config.yml`, or existing verification tasks.
3. If more than one flow is available, prioritize the smallest connected end-to-end slice first.
4. If the run command or target URL is still unclear, ask one question with a recommended default.

### Step 3 — Run browser validation

1. Launch or reuse the app using the documented project command.
2. Exercise the selected user flows in the browser.
3. Capture useful evidence when available:
   - browser-visible failure summary
   - console errors
   - network errors
   - route or auth mismatches with `contract.md`
   - obvious layout or interaction blockers
4. If a flow passes, record the successful path briefly.
5. If a flow fails, stop at the smallest reproducible step and classify the issue.

### Step 4 — Write `browser-report.md`

Write or refresh `FEATURE_DIR/browser-report.md` using this structure:

```markdown
# Browser Validation Report

**Generated by:** kite.browser
**Date:** <ISO-8601 date>

## What this means in plain English

> One short paragraph. Say whether the connected app worked in the browser, what broke first if it did not, and which agent should fix it.

## Flows tested

- <flow name> — pass/fail

## Findings

### <finding title>

- **Class:** [frontend|backend|contract|ops|unknown]
- **Where it failed:** <page, action, or request>
- **Why this matters:** <plain English impact>
- **Evidence:** <console/network/browser summary>
- **Recommended owner:** `kite.frontend` | `kite.backend` | user

## Passed checks

- ...

## Blockers

- ...

## Next recommended command

- `kite.frontend`
```

### Step 5 — Report back to the caller

1. Summarize:
   - number of flows tested
   - number of passes and failures
   - top blocking issue, if any
   - whether the next fix belongs to frontend, backend, contract, ops, or the user

2. Do **not** update `.kite/state.yml`; the parent workflow stage owns state.
